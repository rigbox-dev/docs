# Managed AI Proxy

The managed AI proxy lets your workspace code call supported AI providers without putting provider API keys in the VM. Rigbox holds managed provider keys on your behalf, injects them at request time, and tracks usage against your credit balance.

## How It Works

Every workspace can reach the proxy at `http://172.16.0.1:9090` from inside the VM. When managed mode is active, AI SDK requests can be routed through that bridge-local proxy. The proxy:

1. Intercepts the request from your workspace VM
2. Resolves the workspace from the VM's source IP on the compute bridge
3. Injects the correct provider API key (which never enters your VM)
4. Forwards the request to the upstream provider
5. Logs token usage and deducts from your credit balance
6. Returns the response to your code

Your workspace code uses normal SDKs and HTTP clients. For OpenAI-compatible tools, point the base URL at `/v1`. For provider-native SDKs, use the provider path such as `/google`.

<Tip>
Provider SDK environment variables are opt-in. Rigbox exposes `RIGBOX_AI_PROXY_URL` globally for discovery, and writes `~/.rigbox/proxy.env` for shells or projects that want to route through the proxy.
</Tip>

## Supported Providers

Managed provider availability depends on the keys configured for the current Rigbox environment. Use `rig proxy status` inside a workspace to see the proxy URLs available to your shell.

| Surface | Endpoint | Notes |
|----------|----------|-------|
| **OpenAI-compatible gateway** | `http://172.16.0.1:9090/v1` | Supports `/models`, `/chat/completions`, `/responses`, and `/embeddings` when a managed OpenAI key is configured |
| **Google Gemini provider path** | `http://172.16.0.1:9090/google` | Provider-native Gemini REST shape |
| **Anthropic provider path** | `http://172.16.0.1:9090/anthropic` | Provider-native Anthropic shape when configured |
| **OpenAI provider path** | `http://172.16.0.1:9090/openai` | Legacy/provider-native OpenAI path |
| **Brave Search** | `http://172.16.0.1:9090/brave` | Web Search API when configured |

## Activate Managed Mode

### Via the API

Set the AI configuration for a workspace to managed mode with your chosen provider and model.

<CodeGroup>
```bash cURL
curl -X PUT https://api.rigbox.dev/api/workspaces/{workspace_id}/ai-config \
  -H "Authorization: Bearer $RIGBOX_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "mode": "managed",
    "provider": "google",
    "model": "google/gemini-2.5-pro"
  }'
```

```python Python
import requests

response = requests.put(
    f"https://api.rigbox.dev/api/workspaces/{workspace_id}/ai-config",
    headers={"Authorization": f"Bearer {token}"},
    json={
        "mode": "managed",
        "provider": "google",
        "model": "google/gemini-2.5-pro"
    }
)

print(response.json())
# {"mode": "managed", "provider": "google", "model": "google/gemini-2.5-pro"}
```

```javascript JavaScript
const response = await fetch(
  `https://api.rigbox.dev/api/workspaces/${workspaceId}/ai-config`,
  {
    method: "PUT",
    headers: {
      "Authorization": `Bearer ${token}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      mode: "managed",
      provider: "google",
      model: "google/gemini-2.5-pro",
    }),
  }
);

const config = await response.json();
console.log(config);
```
</CodeGroup>

### Via the CLI (Inside a Workspace VM)

If you are already inside a running workspace, the `rig` CLI can configure your current shell to use the proxy.

```bash
eval "$(rig proxy on)"
```

This command:
- Prints shell `export` statements that set the correct base URLs for each AI SDK
- Uses `managed-by-rigbox` placeholder API keys where SDKs require a non-empty key
- Routes subsequent AI SDK calls in that shell through the Rigbox proxy

<Tip>
Use `rig proxy status` to see available provider URLs, current shell state, and credit balance.
</Tip>

To deactivate:

```bash
eval "$(rig proxy off)"
```

## What Happens When Credits Run Out

When your credit balance reaches zero:

1. **API requests return HTTP 402** - the proxy rejects new managed AI calls until credits are available
2. **The UI shows an upgrade prompt** - you can upgrade to Pro or switch to [BYOK mode](/guides/byok)
3. **Existing services keep running** - only new AI API calls are blocked; your workspace and non-AI services are unaffected

<Warning>
If you have a long-running agent or automated pipeline, it will start receiving 402 errors once credits are exhausted. Consider monitoring your balance programmatically using the credits endpoint.
</Warning>

## End-to-End Example

Here is a complete flow: activate managed mode, make an AI call from inside the workspace, then check remaining credits.

**Step 1 - Activate managed mode for a workspace:**

```bash
curl -X PUT https://api.rigbox.dev/api/workspaces/ws_abc123/ai-config \
  -H "Authorization: Bearer $RIGBOX_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"mode": "managed", "provider": "google", "model": "gemini-2.5-flash-lite"}'
```

**Step 2 - Inside the workspace VM, configure the shell and make a call:**

```bash
# Apply proxy environment variables
eval "$(rig proxy on)"

curl -sS \
  -H "Content-Type: application/json" \
  -X POST "$RIGBOX_AI_PROXY_URL/google/v1beta/models/gemini-2.5-flash-lite:generateContent" \
  -d '{
    "contents": [
      {
        "parts": [
          {"text": "Explain microVMs in one paragraph."}
        ]
      }
    ]
  }' | jq .
```

**Step 3 - Check remaining credits:**

```bash
curl -s https://api.rigbox.dev/api/users/me/credits \
  -H "Authorization: Bearer $RIGBOX_TOKEN" | jq .
# {"remaining": 1795, "total": 2000, "mode": "managed"}
```

<Note>
Credit deduction happens synchronously - the balance is updated before the proxy returns the response to your code, so the credits endpoint always reflects the latest usage.
</Note>

## Switching Providers

You can change the provider and model at any time without restarting the workspace, as long as the selected managed provider is configured for your environment. The next AI API call will use the new configuration.

```bash
# Switch to OpenAI when the managed OpenAI gateway is configured
curl -X PUT https://api.rigbox.dev/api/workspaces/ws_abc123/ai-config \
  -H "Authorization: Bearer $RIGBOX_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"mode": "managed", "provider": "openai", "model": "gpt-4o"}'
```

If you are using the CLI inside the VM, run `eval "$(rig proxy on)"` again to refresh the environment variables for the new provider.
