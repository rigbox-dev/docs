# The Rigbox CLI

The `rig` CLI (alias for `rigbox`) gives you full control over your Rigbox workspaces, apps, and AI credits from the terminal. It is pre-installed in every workspace VM, and you can also install it on your local machine.

---

## Installation

### Inside a workspace

The CLI is already installed in every Rigbox workspace. Open a terminal and run:

```bash
rig --version
```

Inside a workspace, `rig` automatically switches into workspace-scoped mode. You do not need to log in, and account-wide commands are intentionally hidden.

### On your local machine

Install with one command:

```bash
curl -fsSL https://rigbox.dev/install.sh | bash
```

The installer detects your OS and CPU, downloads the latest release, and installs both `rig` and `rigbox` (same binary) to `$HOME/.local/bin`. Two optional environment variables tune behaviour:

| Variable | Purpose | Default |
| --- | --- | --- |
| `RIG_INSTALL_DIR` | Install location for the binary. | `$HOME/.local/bin` |
| `RIG_VERSION` | Pin to a specific release tag instead of `latest`. | `latest` |

```bash
# Install to /usr/local/bin and pin a version
RIG_INSTALL_DIR=/usr/local/bin RIG_VERSION=v0.3.4 \
  bash -c "curl -fsSL https://rigbox.dev/install.sh | bash"
```

Confirm the install with:

```bash
rig --version
```

## Authentication and Runtime Modes

The CLI has two modes depending on where it runs.

| Context | Auth method | Command surface |
|---------|-------------|-----------------|
| Inside a workspace VM | Ambient workspace identity from the VM bridge | Workspace-local commands only |
| On your local machine | OAuth token via `rig login` | Full account and workspace control |

### Outside a workspace

Before using the CLI on your laptop or another external machine, log in:

```bash
rig login
```

This opens your browser for OAuth authentication and stores your token locally.

<Note>
  External CLI configuration is stored in `~/.rigbox/config.json`. This includes your auth token and default settings.
</Note>

### Inside a workspace

Inside a workspace, `rig` detects `/etc/rigbox/workspace.json` and talks to the VM-local bridge API at `http://172.16.0.1:9090/rig/v1`.

No bearer token is sent for normal workspace-scoped operations. The bridge resolves your workspace from the VM's source IP on the compute node, so editing local config files cannot switch the workspace identity.

Run:

```bash
rig whoami
```

The output should show `workspace_ip` or equivalent bridge identity as the auth method.

## Workspace Management

Workspaces are isolated micro-VMs where your code runs. On your local machine, the CLI provides full lifecycle management across your account.

<Note>
  Workspace lifecycle commands such as `rig ls`, `rig new`, `rig spawn`, `rig start`, `rig stop`, `rig restart`, and `rig rm` are external-mode commands. Inside a workspace VM, `rig` is scoped to the current workspace and these account-wide commands are not shown in `rig -h`.
</Note>

<Note>
  Placeholders below use the form `<WORKSPACE_NAME>` and `<APP_NAME>` to make it obvious where you should substitute your own values. Replace the whole placeholder, including the angle brackets, with the actual name (e.g. `my-project`). The walkthrough examples that follow use `my-project` and `my-api` as illustrative names.
</Note>

### List workspaces

```bash
rig ls
```

Output:

```
NAME          STATUS    IMAGE    CREATED
my-project    running   dev      2 hours ago
api-service   stopped   base     3 days ago
ml-notebook   running   dev      1 day ago
```

### Create a workspace

```bash
rig new <WORKSPACE_NAME> --image dev
```

This creates a workspace but does not start it. Available images include `base` (minimal) and `dev` (includes development tools, sidecar, and more).

### Create and start in one command

```bash
rig spawn <WORKSPACE_NAME>
```

<Tip>
  Use `rig spawn` for the fastest path from zero to a running workspace. It creates the workspace, starts the VM, and waits until it is ready - all in one command.
</Tip>

### Start and stop

`rig start`, `rig stop`, `rig restart`, `rig logs`, and `rig rm` accept either a workspace name or a workspace ID (`ws-...`). The placeholder `<WORKSPACE_NAME_OR_ID>` reflects that you can paste either.

```bash
rig start <WORKSPACE_NAME_OR_ID>     # Boot the VM
rig stop <WORKSPACE_NAME_OR_ID>      # Shut down the VM (preserves disk)
```

### Check status

```bash
rig status
```

This shows a dashboard with your account quota, resource usage across all workspaces, and current plan limits.

### Stream logs

```bash
rig logs <WORKSPACE_NAME_OR_ID> --follow
```

Streams workspace system logs in real time. Press `Ctrl+C` to stop.

### Get the SSH command

```bash
rig ssh-info my-project
```

Prints the canonical `ssh user@host` command for a workspace, plus the underlying user, region host, and status. Use `--format json` for scripting.

The output uses region-direct routing (e.g. `eu-west-1.rigbox.dev`) — see the [SSH Access guide](/guides/ssh-access) for the full connection model.

### Delete a workspace

```bash
rig rm <WORKSPACE_NAME_OR_ID>
```

<Warning>
  Deleting a workspace permanently destroys the VM and its disk. This action cannot be undone. Make sure to back up any important files first.
</Warning>

## App Management

Apps are services running inside your workspace that are exposed to the internet via a unique subdomain (`{name}.rigbox.dev`).

Inside a workspace VM, app commands are always scoped to the current workspace. Passing another workspace ID does not change the target workspace.

### List apps

```bash
rig app ls
```

Output:

```
NAME       PORT    STATUS    URL
my-api     3000    running   my-api.rigbox.dev
frontend   5173    running   frontend.rigbox.dev
```

### Create an app

```bash
rig app new <APP_NAME> --port 3000
```

This creates an app route that maps a public subdomain to the specified port inside your workspace. The app will be reachable at `<APP_NAME>.rigbox.dev`.

Pass `--kind cli` for interactive tools you'll invoke over SSH rather than expose as a public service — CLI-shape apps have no port, no protocol, and no Caddy route. Defaults to `service`. The same field is available on the API as `app_kind: "service" | "cli"`; see [Catalog Apps](/guides/catalog#app-shapes) for the full Service-vs-CLI breakdown.

### Start, stop, and restart

`rig app start`, `rig app stop`, `rig app restart`, `rig app logs`, and `rig app rm` accept either an app name or an app ID (`app-...`).

```bash
rig app start <APP_NAME_OR_ID>      # Start the app service
rig app stop <APP_NAME_OR_ID>       # Stop the app service
rig app restart <APP_NAME_OR_ID>    # Restart the app service
```

### Stream app logs

```bash
rig app logs <APP_NAME_OR_ID> --follow
```

### Delete an app

```bash
rig app rm <APP_NAME_OR_ID>
```

## Templates and Catalog

Templates, catalog browsing, and account-level tool management are available from the external CLI. They are hidden inside workspace VM mode.

### Browse templates

Templates are pre-configured workspace setups for common use cases:

```bash
rig template
```

Output:

```
NAME                  DESCRIPTION
ai-agent-starter      AI agent with tool use and memory
openclaw              Telegram bot with AI backend
web-fullstack         React + Node.js full-stack starter
python-ml             Python ML workspace with Jupyter
```

### Browse installable apps

The catalog lists applications you can install into a running workspace:

```bash
rig catalog
```

### Install from catalog

```bash
rig catalog install vscode
```

This installs VS Code Server into your current workspace, making it accessible via a browser.

<Tip>
  The catalog includes popular tools like VS Code, Jupyter, and database clients. Run `rig catalog` to see the full list.
</Tip>

### Install an AI coding CLI

The catalog also ships **CLI-shape recipes** for terminal-first coding agents — `claude`, `codex`, `opencode`, and `kilocode`. These install the binary into `$PATH` and route through the managed AI proxy; there is no port, subdomain, or systemd service:

```bash
rig catalog install claude
ssh my-project-ctbxi8lb@eu-west-1.rigbox.dev
# inside the workspace
claude
```

See [Catalog Apps](/guides/catalog) for the full list of Service- and CLI-shape recipes and the differences between them.

## Tools

Tools are interactive applications that run alongside your workspace.

### List available tools

```bash
rig tool ls <WORKSPACE_NAME_OR_ID>
```

### Launch a tool

```bash
rig tool launch architecture-explorer <WORKSPACE_NAME_OR_ID>
```

This starts the specified tool and provides you with the URL to access it.

## SSH Key Management

Manage the SSH keys registered to your account directly from the external CLI. SSH key commands are account-wide and are not shown inside workspace VM mode.

### Add a key

```bash
rig ssh-key add <KEY_NAME>
```

This reads your default public key (`~/.ssh/id_ed25519.pub`) and registers it with your account under the name you provide (for example, `laptop` or `desktop`).

### List keys

```bash
rig ssh-key ls
```

Output:

```
NAME      FINGERPRINT                               CREATED
laptop    SHA256:abc123def456...                     2 days ago
desktop   SHA256:xyz789ghi012...                     1 week ago
```

### Remove a key

```bash
rig ssh-key rm <KEY_NAME>
```

For more details on SSH access, including connecting to workspaces and using the TUI dashboard, see the [SSH Access guide](/guides/ssh-access).

## AI Proxy

The managed AI proxy lets you use supported AI providers through Rigbox's credit system without putting provider keys in your workspace. See [Managed AI Proxy](/guides/managed-proxy) for the full architecture.

### Default environment

Managed workspaces export OpenRouter-shape variables to every login shell:

```bash
export OPENROUTER_BASE_URL='http://172.16.0.1:9090/v1'
export OPENROUTER_API_KEY='managed-by-rigbox'   # opaque placeholder; real key never enters the VM
```

Most modern AI SDKs and CLIs read these directly. The catalog recipes for `claude`, `codex`, `opencode`, and `kilocode` each layer a `/etc/profile.d/*-routing.sh` shim on top that translates these into the provider-native env vars each agent prefers.

### Configure the current shell

If you need provider-native exports for tools that don't read OpenRouter env vars (Gemini CLI, Aider, etc.), run:

```bash
eval "$(rig proxy on)"
```

This prints compatibility `export` statements (`OPENAI_BASE_URL`, `ANTHROPIC_BASE_URL`, `GOOGLE_API_KEY`, …) wired to the same proxy. You can also source the generated env file directly:

```bash
source ~/.rigbox/proxy.env
```

These compatibility variables are opt-in to avoid hijacking tools you authenticate separately.

### Deactivate the proxy

```bash
eval "$(rig proxy off)"
```

### Check proxy status

```bash
rig proxy status
```

This shows the bridge URL, available provider base URLs, whether your current shell environment matches them, and your credit balance when available.

You can also check credits through the public API from outside the VM:

```bash
curl https://api.rigbox.dev/api/v1/users/me/credits \
  -H "Authorization: Bearer YOUR_API_KEY"
```

<Note>
  The proxy bridge is only reachable from inside a workspace VM. It listens at `http://172.16.0.1:9090`, resolves the workspace by bridge source IP, injects managed provider keys upstream, and meters usage against your credits.
</Note>

## Workspace VM Command Surface

Inside a workspace VM, `rig` is intentionally smaller. Run `rig -h` in the VM to see only workspace-safe commands.

Shown in VM mode:

- `rig whoami`
- `rig status`
- `rig app ...`
- `rig proxy ...`

Hidden in VM mode:

- `rig login` and `rig logout`
- `rig ls`, `rig new`, `rig spawn`, `rig start`, `rig stop`, `rig restart`, and `rig rm`
- `rig template`, `rig catalog`, `rig tool`, and `rig ssh-key`

Those commands remain available from your local machine after `rig login`.
