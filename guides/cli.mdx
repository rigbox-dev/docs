# The Rigbox CLI

The `rig` CLI (alias for `rigbox`) gives you full control over your Rigbox workspaces, apps, and AI credits from the terminal. It is pre-installed in every workspace VM, and you can also install it on your local machine.

---

## Installation

### Inside a workspace

The CLI is already installed in every Rigbox workspace. Open a terminal and run:

```bash
rig --version
```

Inside a workspace, `rig` automatically switches into workspace-scoped mode. You do not need to log in, and account-wide commands are intentionally hidden.

### On your local machine

Download the latest release for your platform:

<CodeGroup>

```bash macOS (Apple Silicon)
curl -fsSL https://github.com/rigbox-dev/cli/releases/latest/download/rigbox-darwin-arm64 -o /usr/local/bin/rig
chmod +x /usr/local/bin/rig
```

```bash macOS (Intel)
curl -fsSL https://github.com/rigbox-dev/cli/releases/latest/download/rigbox-darwin-amd64 -o /usr/local/bin/rig
chmod +x /usr/local/bin/rig
```

```bash Linux (x86_64)
curl -fsSL https://github.com/rigbox-dev/cli/releases/latest/download/rigbox-linux-amd64 -o /usr/local/bin/rig
chmod +x /usr/local/bin/rig
```

</CodeGroup>

## Authentication and Runtime Modes

The CLI has two modes depending on where it runs.

| Context | Auth method | Command surface |
|---------|-------------|-----------------|
| Inside a workspace VM | Ambient workspace identity from the VM bridge | Workspace-local commands only |
| On your local machine | OAuth token via `rig login` | Full account and workspace control |

### Outside a workspace

Before using the CLI on your laptop or another external machine, log in:

```bash
rig login
```

This opens your browser for OAuth authentication and stores your token locally.

<Note>
  External CLI configuration is stored in `~/.rigbox/config.json`. This includes your auth token and default settings.
</Note>

### Inside a workspace

Inside a workspace, `rig` detects `/etc/rigbox/workspace.json` and talks to the VM-local bridge API at `http://172.16.0.1:9090/rig/v1`.

No bearer token is sent for normal workspace-scoped operations. The bridge resolves your workspace from the VM's source IP on the compute node, so editing local config files cannot switch the workspace identity.

Run:

```bash
rig whoami
```

The output should show `workspace_ip` or equivalent bridge identity as the auth method.

## Workspace Management

Workspaces are isolated Firecracker micro-VMs where your code runs. On your local machine, the CLI provides full lifecycle management across your account.

<Note>
  Workspace lifecycle commands such as `rig ls`, `rig new`, `rig spawn`, `rig start`, `rig stop`, `rig restart`, and `rig rm` are external-mode commands. Inside a workspace VM, `rig` is scoped to the current workspace and these account-wide commands are not shown in `rig -h`.
</Note>

### List workspaces

```bash
rig ls
```

Output:

```
NAME          STATUS    IMAGE    CREATED
my-project    running   dev      2 hours ago
api-service   stopped   base     3 days ago
ml-notebook   running   dev      1 day ago
```

### Create a workspace

```bash
rig new my-project --image dev
```

This creates a workspace but does not start it. Available images include `base` (minimal) and `dev` (includes development tools, sidecar, and more).

### Create and start in one command

```bash
rig spawn my-project
```

<Tip>
  Use `rig spawn` for the fastest path from zero to a running workspace. It creates the workspace, starts the VM, and waits until it is ready - all in one command.
</Tip>

### Start and stop

```bash
rig start my-project     # Boot the VM
rig stop my-project      # Shut down the VM (preserves disk)
```

### Check status

```bash
rig status
```

This shows a dashboard with your account quota, resource usage across all workspaces, and current plan limits.

### Stream logs

```bash
rig logs my-project --follow
```

Streams workspace system logs in real time. Press `Ctrl+C` to stop.

### Delete a workspace

```bash
rig rm my-project
```

<Warning>
  Deleting a workspace permanently destroys the VM and its disk. This action cannot be undone. Make sure to back up any important files first.
</Warning>

## App Management

Apps are services running inside your workspace that are exposed to the internet via a unique subdomain (`{name}.rigbox.dev`).

Inside a workspace VM, app commands are always scoped to the current workspace. Passing another workspace ID does not change the target workspace.

### List apps

```bash
rig app ls
```

Output:

```
NAME       PORT    STATUS    URL
my-api     3000    running   my-api.rigbox.dev
frontend   5173    running   frontend.rigbox.dev
```

### Create an app

```bash
rig app new my-api --port 3000
```

This creates an app route that maps a public subdomain to the specified port inside your workspace.

### Start, stop, and restart

```bash
rig app start my-api      # Start the app service
rig app stop my-api       # Stop the app service
rig app restart my-api    # Restart the app service
```

### Stream app logs

```bash
rig app logs my-api --follow
```

### Delete an app

```bash
rig app rm my-api
```

## Templates and Catalog

Templates, catalog browsing, and account-level tool management are available from the external CLI. They are hidden inside workspace VM mode.

### Browse templates

Templates are pre-configured workspace setups for common use cases:

```bash
rig template
```

Output:

```
NAME                  DESCRIPTION
ai-agent-starter      AI agent with tool use and memory
openclaw              Telegram bot with AI backend
web-fullstack         React + Node.js full-stack starter
python-ml             Python ML workspace with Jupyter
```

### Browse installable apps

The catalog lists applications you can install into a running workspace:

```bash
rig catalog
```

### Install from catalog

```bash
rig catalog install vscode
```

This installs VS Code Server into your current workspace, making it accessible via a browser.

<Tip>
  The catalog includes popular tools like VS Code, Jupyter, and database clients. Run `rig catalog` to see the full list.
</Tip>

## Tools

Tools are interactive applications that run alongside your workspace.

### List available tools

```bash
rig tool ls my-project
```

### Launch a tool

```bash
rig tool launch architecture-explorer my-project
```

This starts the specified tool and provides you with the URL to access it.

## SSH Key Management

Manage the SSH keys registered to your account directly from the external CLI. SSH key commands are account-wide and are not shown inside workspace VM mode.

### Add a key

```bash
rig ssh-key add laptop
```

This reads your default public key (`~/.ssh/id_ed25519.pub`) and registers it with your account.

### List keys

```bash
rig ssh-key ls
```

Output:

```
NAME      FINGERPRINT                               CREATED
laptop    SHA256:abc123def456...                     2 days ago
desktop   SHA256:xyz789ghi012...                     1 week ago
```

### Remove a key

```bash
rig ssh-key rm laptop
```

For more details on SSH access, including connecting to workspaces and using the TUI dashboard, see the [SSH Access guide](/guides/ssh-access).

## AI Proxy

The managed AI proxy lets you use supported AI providers through Rigbox's credit system without putting provider keys in your workspace.

### Configure the current shell

From inside a workspace VM:

```bash
eval "$(rig proxy on)"
```

This prints and applies SDK environment variables for the current shell:

```bash
export RIGBOX_AI_PROXY_URL='http://172.16.0.1:9090'
export OPENAI_BASE_URL='http://172.16.0.1:9090/v1'
export OPENAI_API_KEY='managed-by-rigbox'
export GEMINI_BASE_URL='http://172.16.0.1:9090/google'
export GOOGLE_API_KEY='managed-by-rigbox'
```

You can also source the generated env file directly:

```bash
source ~/.rigbox/proxy.env
```

Provider SDK variables are opt-in. Rigbox does not globally export `OPENAI_BASE_URL`, `ANTHROPIC_BASE_URL`, or `GEMINI_BASE_URL` for every shell session, because that can hijack other tools that you authenticate separately.

### Deactivate the proxy

```bash
eval "$(rig proxy off)"
```

### Check proxy status

```bash
rig proxy status
```

This shows the bridge URL, available provider base URLs, whether your current shell environment matches them, and your credit balance when available.

You can also check credits through the public API from outside the VM:

```bash
curl https://api.rigbox.dev/api/users/me/credits \
  -H "Authorization: Bearer YOUR_API_KEY"
```

<Note>
  The proxy bridge is only reachable from inside a workspace VM. It listens at `http://172.16.0.1:9090`, resolves the workspace by bridge source IP, injects managed provider keys upstream, and meters usage against your credits.
</Note>

## Workspace VM Command Surface

Inside a workspace VM, `rig` is intentionally smaller. Run `rig -h` in the VM to see only workspace-safe commands.

Shown in VM mode:

- `rig whoami`
- `rig status`
- `rig app ...`
- `rig proxy ...`

Hidden in VM mode:

- `rig login` and `rig logout`
- `rig ls`, `rig new`, `rig spawn`, `rig start`, `rig stop`, `rig restart`, and `rig rm`
- `rig template`, `rig catalog`, `rig tool`, and `rig ssh-key`

Those commands remain available from your local machine after `rig login`.
